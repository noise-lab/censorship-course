# Lecture 3 — Speaker Notes (Throttling + DoS)

Per-slide context + clickable links. In-deck notes (the `::: {.notes}` blocks) are also visible via reveal.js speaker view (press **S**). Every URL was verified this session or is a canonical org landing page.

| Slide | Context + Links |
|---|---|
| **A Tax on Access, Not a Wall** | The "friction is a form of control" frame is from Roberts, [*Censored*](https://press.princeton.edu/books/hardcover/9780691178868/censored) (Princeton, 2018). Book context: §2.3, "Disrupting the Flow of Data." |
| **Two Ways to Degrade, Not Block** | Both are *friction* mechanisms — no block page, no ban. If a student asks "why isn't this outright blocking?" — the book defines information control as any *tax* on access. |
| **Throttling Is Friction by Design** | Rate limiting is universally dual-use — every prod system does it for fairness / SLAs / DoS protection. The censorship use is the same mechanism aimed at particular targets. Book §2.3.1. |
| **How a Traffic Shaper Works** | Identify → shape is the canonical model. If someone wants depth: **Sandvine** is the vendor name to know (marketed to ISPs and states alike; Bloomberg has a good longform — search "Sandvine deep packet inspection Bloomberg"). |
| **Identifying What to Throttle** | SNI-based and DNS-based identification are what QUIC + [DoH](https://datatracker.ietf.org/doc/html/rfc8484) + [TLS ECH](https://datatracker.ietf.org/doc/html/draft-ietf-tls-esni) are designed to defeat — ties directly to Wednesday's Protocols lecture. |
| **Shaping: Leaky Bucket vs. Token Bucket** | CS textbook material. Primers: [Leaky bucket (Wikipedia)](https://en.wikipedia.org/wiki/Leaky_bucket), [Token bucket (Wikipedia)](https://en.wikipedia.org/wiki/Token_bucket). Real deployments use token bucket because burstiness matters for UX. |
| **Why Throttling Is So Hard to Catch** | **Wehe** (Northeastern's throttling detector): <https://wehe.meddle.mobi/>. Paper: Li et al., "A Large-Scale Analysis of Deployed Traffic Differentiation Practices" (SIGCOMM 2019). **Netflix ↔ Comcast 2014** peering-vs-throttling saga: [Comcast–Netflix dispute (Wikipedia)](https://en.wikipedia.org/wiki/Comcast%E2%80%93Netflix_dispute) — foreshadows net neutrality (Ch. 4). |
| **Throttling as a Live Political Tool** | Primary source: [Access Now — Internet shutdowns in 2025](https://www.accessnow.org/internet-shutdowns-2025/) (KeepItOn 2025, released Mar 31 2026). Headline numbers: **313 shutdowns / 52 countries** (up from 296 in 2024); **India** leads at **65**. Turkey throttling around unrest/elections is documented at [OONI Turkey](https://explorer.ooni.org/country/TR). |
| **Throttling at Home: A Private "Tax on Access"** | The story that broke it: [Washington Post, Aug 15 2023 — Twitter throttles links to Threads, Bluesky and NYT](https://www.washingtonpost.com/technology/2023/08/15/twitter-x-links-delayed/). Follow-ups: [Variety](https://variety.com/2023/digital/news/elon-musk-x-twitter-delay-links-meta-new-york-times-1235697091/), [Rappler/Reuters](https://www.rappler.com/technology/social-media/x-delays-access-content-reuters-new-york-times-social-media-rivals/). Mechanism: X used its `t.co` redirect to insert ~5 sec of latency. After the Post ran, NYT/Reuters delays quietly vanished; Bluesky/Substack persisted. |
| **DoS: Exhaust a Finite Resource** | Three-resource taxonomy (network / connections / server) is the book's frame. Distinguish DoS (one source) vs. DDoS (many, usually a botnet). Mirai/IoT made DDoS cheap; deeper botnet coverage in Ch. 3. |
| **Why DoS Is Hard to Defend: Asymmetry** | Asymmetry / spoofing / indistinguishability — the book's teaching triad. Every defense on the "Defending Against DoS" slide attacks one of the three. |
| **SYN Flood: Asymmetry in TCP** | Attack + mitigations: [RFC 4987 — TCP SYN Flooding Attacks and Common Mitigations](https://datatracker.ietf.org/doc/html/rfc4987) (2007). SYN cookies original writeup: [D.J. Bernstein — SYN cookies](https://cr.yp.to/syncookies.html). Summary: [SYN flood (Wikipedia)](https://en.wikipedia.org/wiki/SYN_flood). |
| **Reflection and Amplification** | Canonical case: **Spamhaus, March 2013, ~300 Gbps via open DNS resolvers** — [DDoS attack on Spamhaus (Wikipedia)](https://en.wikipedia.org/wiki/DDoS_attack_on_Spamhaus). Defense: [RFC 2827 / BCP 38 — Ingress Filtering](https://datatracker.ietf.org/doc/html/rfc2827) (Ferguson & Senie, 2000; still under-deployed). Modern rhyme: **GitHub took 1.35 Tbps via memcached (Feb 2018)** — [GitHub incident report](https://github.blog/2018-03-01-ddos-incident-report/) — ~50,000× amplification. |
| **When DoS Becomes Censorship Infrastructure** | Landmark research: Marczak et al., "An Analysis of China's 'Great Cannon'" (FOCI 2015). Citizen Lab page: [China's Great Cannon](https://citizenlab.ca/2015/04/chinas-great-cannon/). Targets: **GitHub** (hosting GreatFire and NYT-China mirrors) and **GreatFire.org**. Mechanism: on-path, co-located with the GFW; injected JavaScript into Baidu responses to turn external browsers into an attack botnet. |
| **Whose Browsers Got Hijacked?** | The Taiwan / Hong Kong concentration chart on this slide is from the Citizen Lab paper. Teaching point: *the weapon was built from innocent bystanders* — anyone outside China loading Baidu-hosted analytics. |
| **Centralization Makes One Target Enough** | **Dyn attack, Oct 21 2016**: [DDoS attack on Dyn (Wikipedia)](https://en.wikipedia.org/wiki/DDoS_attack_on_Dyn). Botnet mechanics: [Mirai (Wikipedia)](https://en.wikipedia.org/wiki/Mirai_(malware)). Academic anchor: Antonakakis et al., ["Understanding the Mirai Botnet"](https://www.usenix.org/conference/usenixsecurity17/technical-sessions/presentation/antonakakis) (USENIX Security 2017). Modern rhyme: [Cloudflare's Nov 18 2025 outage](https://blog.cloudflare.com/18-november-2025-outage/) — same centralization theme, different failure mode (internal misconfig instead of attack). |
| **Defending Against DoS** | Concept students haven't seen: **anycast** — one IP advertised from many locations; BGP delivers to nearest. All 13 DNS root "servers" are anycast constellations. [Anycast (Wikipedia)](https://en.wikipedia.org/wiki/Anycast). Scrubbing services worth naming: Cloudflare Magic Transit, Akamai Prolexic, Radware, Neustar UltraDDoS. Booter/stresser takedowns: [Europol Operation PowerOFF](https://www.europol.europa.eu/media-press/newsroom/news/law-enforcement-shuts-down-27-online-platforms-used-for-launching-ddos-attacks) (Dec 2024, ongoing). |
| **Who Gets to Stay Online?** | The 108.9 B / 241% figures come from Cloudflare's [Project Galileo 11th anniversary post (Jun 2025)](https://blog.cloudflare.com/celebrating-11-years-of-project-galileo-global-impact/) and the [Cloudflare 5th Annual Impact Report (Dec 19 2025)](https://www.cloudflare.com/press/press-releases/2025/cloudflare-publishes-fifth-annual-impact-report/). Two case studies worth naming: **Belarusian Investigative Center** — onboarded Sep 27 2024 mid-attack; 28 B requests the next day. **Tech4Peace** — 12-day attack from Mar 10 2025, 2.7 B requests, adaptive intensity. The tension is the same one from Lecture 1: Cloudflare protects them, but also gets to decide who counts as "public interest" — cf. [Prince's "no one should have that power" post on Daily Stormer](https://blog.cloudflare.com/why-we-terminated-daily-stormer/). |
| **The Takeaway: Degrade, Don't Block** | Roberts's *friction* frame closes the arc. Bridges: **Ch. 5** (measurement — this is *why* OONI, Wehe, Censored Planet, KeepItOn dashboards exist) and **Ch. 4** (peering disputes, net neutrality, Cloudflare-as-arbiter).
